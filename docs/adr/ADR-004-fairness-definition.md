# ADR-2026-05-01-004: 공정성을 "약한 동등 시도 기회를 갖는 선착순"으로 정의

## Status

Accepted

## Context

본 시스템 요구사항에는 두 개의 공정성 관련 키워드가 동시에 등장한다.
- *"**선착순** 예약 시스템 구축"*
- *"모든 사용자가 **동등한 확률**로 상품을 구매할 수 있는 구조를 고민하십시오"*

이 두 키워드는 본질적으로 충돌한다. 선착순(FCFS)은 시간 순서 기반이므로 네트워크 지연, 지리적 위치, 디바이스 성능, 매크로 사용 여부에 따라 사용자 간 성공 확률이 본질적으로 불균등하다. 반면 *"동등한 확률"*의 엄격한 해석은 추첨(Lottery)에 가깝고, 5분 윈도우 내 들어온 모든 요청 중 무작위 N명을 선택하는 형태가 된다.

선착순과 추첨은 단순한 구현 차이가 아니라 **공정성 철학의 차이**다. 콘서트 티켓팅은 관행적으로 선착순, Nike SNKRS의 한정판 운동화는 추첨, 백신 접종 예약은 배정/추첨이다. 이는 도메인별로 *"어떤 종류의 공정성이 사회적으로 기대되는가"*가 다르기 때문이다.

본 시스템의 도메인은 **숙소 예약 + 초특가 프로모션**이다. 일반 숙소 예약은 선착순이 관행이며, 초특가 프로모션이라는 한정 이벤트 성격이 추첨을 정당화할 만큼 도메인 관행을 뒤집을 명분(예: 봇 문제로 진성 팬 보호 절실)이 약하다. 또한 추첨은 결과 발표/통보/예약 확정 등 별도 운영 프로세스를 요구해 구현 복잡도가 크다.

따라서 *"선착순"*을 우선 채택하되, *"동등한 확률"*을 **약한 의미로 재해석**해 ADR로 명시한다. 이는 모순적 요구사항을 의식적으로 다뤘음을 기록하기 위함이다.

## Options Considered

| 옵션 | Pros | Cons | 비용 |
|---|---|---|---|
| A. 엄격한 선착순 (ms 단위 FCFS) | 구현 단순, 도메인 관행 일치 | *"동등한 확률"* 요구사항을 명목적으로만 충족, 매크로 압도적 유리 | 구현 비용 낮음, 신뢰성 비용 ↑ |
| B. 추첨 (5분 receive 후 무작위 N명) | 네트워크 격차 무관, 매크로 영향 ↓, *"동등한 확률"* 엄격 충족 | 결과 통보/환불 등 별도 운영 프로세스 필요, 숙소 예약 도메인 관행과 불일치 | 구현 비용 높음, UX 복잡 |
| C. 하이브리드 (30초 엔트리 윈도우 + 추첨) | ms 단위 차이 무력화, 도메인 관행과 절충 | 30초 윈도우 동안 사용자가 무엇을 보고 기다리는지 UX 모호, 윈도우 시간 정당화 어려움 | 구현 비용 중간, 설계 비용 ↑ |
| D. **선착순 + *"동등한 확률"*의 약한 해석** | 도메인 관행 일치, 운영 단순, 모순 요구사항을 의식적으로 처리 | 절대적 확률 균등은 포기, 매크로/네트워크 격차는 별도 메커니즘으로 보완 필요 | 구현 비용 낮음, 보완 메커니즘 필요 |

## Decision

**옵션 D 채택. 공정성을 *"약한 동등 시도 기회를 갖는 선착순"*으로 정의한다.**

명시적 정의:
- **선착순(FCFS)**: 도착 시각 순서로 ticket을 부여하고 처리한다.
- **약한 동등 확률**: *"절대적 확률 균등"*이 아닌 **"모든 사용자에게 시도 기회 자체는 동등하게 열려 있다"**는 의미로 해석한다. 즉, 시스템이 사용자를 차별하는 게이트(예: 특정 사용자만 우선 입장)는 두지 않는다. 단, 시도 이후의 성공 확률은 네트워크 지연 등 시스템 외 요인의 영향을 받음을 인정한다.

핵심 trade-off:
- **Buy:** 도메인 관행과 일치, 단순한 운영 모델, 명확한 결제 흐름(*"결제 완료 = 예약 성공"*).
- **Pay:** 네트워크 지연/지리/디바이스 성능에 따른 사용자 간 불균등을 시스템적으로 해결하지 않는다. 봇/매크로의 우위를 구조적으로 막지 않는다 — 이 부분은 ADR-005(Rate Limiting)와 운영 환경의 본인인증 가정으로 보완한다.

옵션 B(추첨)를 선택하지 않은 이유: 본 시스템의 도메인이 추첨을 정당화할 만큼 강한 명분(SNKRS급 봇 문제, 백신급 사회적 형평성)을 갖지 않는다. 또한 추첨은 발표/통보/예약 확정의 비동기 운영 프로세스를 요구해, 결제 완료 = 예약 성공의 동기 모델을 깨뜨린다.

옵션 C(하이브리드)를 선택하지 않은 이유: 30초 엔트리 윈도우의 적정 길이를 정당화하기 어렵고, 사용자가 그 30초 동안 무엇을 보며 기다리는지의 UX 설계가 모호하다.

## Consequences

**긍정 결과**
- 도메인 관행과 사용자 기대치가 일치해 별도 사용자 교육이 불필요하다.
- *"결제 완료 = 예약 성공"*의 동기 모델이 유지되어 결제/예약/취소 흐름이 단순하다.
- 모순적 요구사항을 명시적으로 다룬 기록이 남아, 향후 정책 변경 시 출발점이 된다.

**부정 결과**
- 5G 사용자 vs LTE 사용자, 서울 사용자 vs 지방 사용자 사이에 측정 가능한 성공 확률 차이가 발생한다. 이는 시스템 외 요인이지만 사용자 입장에서는 *"공정하지 않다"*는 인식을 가질 수 있다.
- 봇/매크로가 동시 다발 요청으로 큐의 상당수를 점유할 가능성이 존재한다. ADR-005의 Rate Limit이 단일 식별자 기준 비정상 행동만 차단하므로, 분산 봇 공격은 본인인증 계층에서 막아야 한다.
- *"동등한 확률"*의 엄격 해석을 우선시하는 평가자/이해관계자 입장에서는 본 결정이 요구사항을 약화 해석한 것으로 비칠 수 있다.

**재검토 시점**
- 매크로/봇 트래픽이 큐의 50% 이상을 차지하는 패턴이 관찰될 때 (옵션 C 또는 B로 전환 검토)
- 도메인이 변경될 때 (예: 일반 숙소 → 한정판 콘서트 패키지 등 진성 팬 보호가 핵심 가치가 되는 도메인)
- *"동등한 확률"*의 사회적/법적 해석이 강화될 때 (예: 디지털 격차 관련 규제 도입)
