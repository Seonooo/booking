# ADR-2026-05-01-006: 멱등성 키 기반 중복 결제 방지 — 클라이언트 발급 키 + 서버 body 검증 + Redis/DB 이중 계층

## Status

Accepted

## Context

본 시스템 요구사항은 다음을 명시한다.
> "주문서에서 아주 짧은 간격으로 연속되어 결제요청이 되어도 중복처리가 되지 않도록 멱등성 처리가 적용되어야 합니다."

결제는 정합성이 깨질 경우 즉각 금전 손실로 이어지는 도메인이다. 사용자가 "결제하기" 버튼을 더블클릭하거나, 네트워크 지연으로 클라이언트가 자동 재시도하거나, 페이지를 새로고침하는 일상적 행동이 이중 결제로 이어져선 안 된다. 동시에 동일 금액을 별도 결제로 의도하는 정상 케이스(예: 가족 인원수만큼 분할 결제)도 차단되어선 안 된다.

멱등성 처리에는 네 개의 결정 축이 있다.
1. **키 발급 주체**: 클라이언트 vs 서버
2. **저장소**: 휘발성(Redis) vs 영속성(DB unique) vs 이중 계층
3. **TTL**: 결제 흐름과 정합되는 적정 시간
4. **응답 정책**: 중복 요청에 대한 상태 코드 분리

또한 본 시스템은 ADR-001(큐 길이 30, 결제 TTL 5분)과 정합성을 가져야 한다. 멱등성 키의 수명이 결제 TTL과 어긋나면, 사용자 재시도 흐름과 키 만료 흐름이 충돌해 사용자 혼란을 야기한다.

## Options Considered

### 축 1. 키 발급 주체

| 옵션 | Pros | Cons |
|---|---|---|
| A. 서버가 키 도출 (해시 기반) | 클라이언트 단순 | 같은 사용자가 같은 상품을 의도적으로 여러 번 결제하는 정상 케이스 구분 불가 |
| **B. 클라이언트가 키 발급 + 서버 검증** | Stripe/AWS API 표준, 정상 재구매 케이스 자연 지원, 변조 검증 가능 | 클라이언트가 UUID 생성 책임 |

### 축 2. 저장소

| 옵션 | Pros | Cons |
|---|---|---|
| A. Redis 단일 | 빠름, 단순 | Redis 장애 시 멱등성 보장 깨짐 |
| B. DB unique constraint 단일 | 강한 정합성 | 매 요청마다 DB INSERT 시도, 1000 TPS에서 부하 |
| **C. Redis + DB 이중 계층** | Redis로 1차 빠른 차단 + DB unique로 최종 정합성 | 구현 복잡도 ↑, 두 저장소 일관성 관리 필요 |

### 축 3. TTL

| 옵션 | Pros | Cons |
|---|---|---|
| A. 24시간 (관행) | 사용자가 하루 내 어떤 시점에 재시도해도 보호 | 결제 TTL(5분)과 어긋남 → 5분 후 새 결제로 처리되는 시나리오에서 사용자 혼란 |
| **B. 결제 TTL + 마진 = 15분** | 결제 흐름과 정합, race condition 방어 마진 포함 | 15분 후 재시도는 새 결제로 처리됨 (정상 동작) |
| C. 5분 (결제 TTL과 동일) | 가장 정합적 | 마진 없어 race condition 가능 |

### 축 4. 응답 정책 — 동시성 시나리오별

| 시나리오 | 옵션 A: 단일 응답 | 옵션 B: 3-state 분리 |
|---|---|---|
| 같은 키 + 같은 body, 첫 요청 처리 중 | 200 또는 409 | **409 Conflict** ("처리 중, 잠시 후") |
| 같은 키 + 같은 body, 이미 완료 | 캐시 응답 또는 충돌 | **200 OK + 캐시 응답** |
| 같은 키 + 다른 body | 거부 | **422 Unprocessable Entity** ("데이터 충돌, 키 재발급") |

## Decision

**축 1: 옵션 B — 클라이언트가 키 발급 + 서버 body 검증**
**축 2: 옵션 C — Redis + DB unique constraint 이중 계층**
**축 3: 옵션 B — TTL 15분 (결제 TTL 5분 + 마진 10분)**
**축 4: 옵션 B — 3-state 분리 (409 / 422 / 200)**

### 흐름

```
1. GET /checkout 호출 시 서버가 idempotency_key 발급해 응답에 포함
   (클라이언트가 직접 UUID v4 생성하는 것도 허용)

2. POST /booking 요청
   요청: { idempotency_key, userId, productId, amount, paymentMethod, points, ... }

3. 서버 처리 (Lua script 내 atomic):
   a) Redis에서 idempotency_key 조회
   b) 없음 → Redis에 "PROCESSING" 상태로 SETNX (TTL 15분), DB Booking 테이블 INSERT 시도
   c) 있음 + 상태 PROCESSING → 핵심 필드 해시 비교
      - 일치 → 409 Conflict ("처리 중")
      - 불일치 → 422 Unprocessable Entity
   d) 있음 + 상태 COMPLETED → 핵심 필드 해시 비교
      - 일치 → 200 OK + 캐시된 응답
      - 불일치 → 422 Unprocessable Entity

4. 결제 완료 후 Redis 상태를 "COMPLETED"로 갱신 + 응답 캐시
```

### 핵심 필드 정의

해시 비교에 사용되는 "핵심 필드"는 **결제 결과 또는 자원 차감에 영향을 주는 모든 필드**로 정의한다.

| 필드 | 분류 | 비고 |
|---|---|---|
| userId | 핵심 | 결제 주체 |
| productId | 핵심 | 결제 대상 |
| amount | 핵심 | 결제 금액 |
| paymentMethod | 핵심 | card / ypay / point — 차감 자원 종류 결정 |
| points | 핵심 | 사용 포인트 — 차감 자원 |
| timestamp | 부수 | 재시도마다 변동, 결과 영향 없음 |
| request_id | 부수 | 클라이언트 추적용 메타 |
| client_version, user_agent | 부수 | 메타 정보 |

```
hash = SHA256(userId + productId + amount + paymentMethod + points)
```

### 핵심 trade-off

- **Buy:** 이중 결제 결정적 차단 (Redis 장애 시에도 DB unique constraint가 최종 방어선). 클라이언트의 더블클릭/네트워크 재시도/새로고침을 자연스럽게 흡수. 변조 시도(같은 키로 다른 body)를 명시적으로 거부.
- **Pay:** Redis와 DB 두 저장소의 일관성 관리 부담. 매 결제마다 DB INSERT 1회 추가(1000 TPS 환경에서는 큐 길이 30 cap이 있어 실제 DB INSERT는 분당 30회 수준 — 부담 미미). 클라이언트가 UUID 생성/관리 책임.

옵션 1A(서버 키 도출)를 선택하지 않은 이유: 같은 사용자의 같은 상품에 대한 정상 다중 결제(가족 분할 결제 등)를 시스템이 차단하게 됨. 도메인이 향후 이런 케이스를 허용해야 할 수 있음.

옵션 2A(Redis 단일)를 선택하지 않은 이유: 결제 도메인은 Redis 장애 시에도 이중결제가 발생해선 안 되는 영역. Redis 장애를 가정한 ADR-007(예정)과의 정합성을 위해 영속 계층 필수.

옵션 2B(DB 단일)를 선택하지 않은 이유: 1000 TPS에서 매 요청 DB INSERT는 부담. Redis 1차 차단으로 DB INSERT를 큐 통과한 30건 이내로 제한.

## Consequences

**긍정 결과**
- 사용자의 더블클릭, 네트워크 재시도, 새로고침으로 인한 이중 결제가 결정적으로 차단된다.
- Redis 장애 상황에서도 DB unique constraint가 최종 방어선으로 작동해 이중 결제가 발생하지 않는다.
- 응답 코드 3-state(409/422/200)로 클라이언트가 상황을 명확히 구분 가능하다. 특히 422는 변조 시도를 명시해 보안 모니터링 대상으로 활용 가능.
- 결제 TTL(5분)과 멱등성 TTL(15분)의 정합성으로 사용자 재시도 흐름 자연스러움.

**부정 결과**
- Redis와 DB 두 저장소 사이의 일관성 관리가 필요하다. 예: Redis는 INSERT 됐는데 DB INSERT가 실패한 경우 수동 정리 또는 보상 로직 필요.
- 멱등성 키 TTL 15분 이후 재시도는 "새 결제"로 처리된다. 사용자가 30분 뒤에 같은 키로 재시도하는 비정상 케이스는 방어되지 않음(현실적으로 문제 없는 시나리오).
- DB Booking 테이블에 `idempotency_key UNIQUE` 인덱스 추가로 INSERT 비용 약간 증가 (현 트래픽에서는 무시 가능).
- 클라이언트가 멱등성 키 생성 후 임의 변경(예: 쿠폰 자동 제거)을 하면 422가 발생한다. 클라이언트 측에서 "키 발급 후 핵심 필드 변경 금지" 정책을 지켜야 함.

**재검토 시점**
- DB Booking 테이블 INSERT 부하가 임계값을 넘을 때 (DB 단일 계층 또는 분산 캐시로 전환 검토)
- Redis-DB 불일치 사례가 운영 중 관찰될 때 (보상 로직 강화 또는 outbox 패턴 도입 검토)
- 멱등성 TTL 만료 직후 재시도가 빈번히 관찰될 때 (TTL 연장 검토)

## Out of Scope

다음 항목은 현재 스펙에 포함되지 않으나, 추가 시 본 ADR의 핵심 필드 정의를 확장해야 한다. 추가 시 별도 ADR로 본 ADR을 supersede하거나 amendment를 추가할 것.

### 1. 쿠폰 (coupon, promotionId)

현재 스펙에 쿠폰은 포함되지 않는다. 그러나 쿠폰은 **차감 자원**이므로 향후 추가 시 핵심 필드에 반드시 포함되어야 한다.

근거: 동일 amount라도 쿠폰 사용 여부에 따라 결제 의도와 차감 자원이 다르다.
```
요청 1: 정가 60000 - 쿠폰 10000 = amount 50000, coupon: "SUMMER20"
요청 2: 정가 50000 = amount 50000, coupon: null

amount는 같지만 차감 자원(쿠폰 1회 사용권)이 달라 별개 결제로 다뤄져야 함.
```

쿠폰을 부수 필드로 분류하면, 첫 결제 후 쿠폰 만료 또는 자동 제거 시 두 번째 요청이 첫 응답을 그대로 반환받아 정합성이 깨진다.

### 2. 그 외 향후 확장 가능 차감 자원

- 멤버십 등급 할인
- 적립금/마일리지 외 별도 토큰
- 프로모션 코드

위 항목들이 추가되면 모두 **차감 자원**이므로 핵심 필드 포함 필수.

### 3. 클라이언트 측 키 관리 정책

클라이언트가 멱등성 키 발급 후 핵심 필드(특히 쿠폰)를 임의 변경하지 않도록 하는 프론트엔드 정책은 본 ADR의 범위 밖이다. 백엔드는 422 응답으로 변경을 거부할 뿐이며, UX 가이드는 별도 정의되어야 한다.
