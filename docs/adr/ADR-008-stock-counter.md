# ADR-2026-05-01-008: 재고 카운터 기반 단일 진입 모델 — 풀린 재고는 새로고침 폴링으로 재참여

## Status

Accepted — supersedes [ADR-001](./ADR-001-queue-length.md)

## Context

ADR-001에서는 *"큐 길이 30(재고의 3배) + 결제 실패 시 큐 내 자동 이전"* 모델을 채택했다. 이후 토론에서 본 모델이 갖는 두 가지 한계가 제기되었다.

첫째, 큐 길이 N=30이라는 숫자가 추정 결제 실패율에 의존해 정당화가 약했다. 결제 실패율의 직접 통계는 한국 PG사들이 공개하지 않으며, 운영 데이터를 확보하기 전까지는 가정에 의존한다.

둘째, 한국 예매 서비스의 실무 관행(인터파크 등)은 *"재고 = 진입"*의 단일 모델을 채택한다. 좌석/재고 hold 후 5~7분 TTL 만료 시 풀린 자리를 사용자의 새로고침 폴링으로 재참여하는 구조다. *"취켓팅"*이라는 산업 용어가 바로 이 모델의 산물이며, 사용자도 이 흐름에 익숙하다.

셋째, ADR-004(약한 동등 시도 기회)의 정의 — *"모든 사용자에게 시도 기회 자체는 동등하게 열려 있다"* — 와의 정합성 측면에서, 큐 30석 모델은 자정 직후의 30명에게만 결제 자격을 주고 그 후 도착자에게는 구조적으로 기회를 막는다. 새 모델은 5분 윈도우 내내 모든 사용자가 폴링으로 재참여 가능하다.

따라서 모델을 다음과 같이 재정의한다.
- 진입 = 재고 atomic DECR
- 결제 실패/TTL 만료 = 재고 atomic INCR
- 풀린 재고 = 사용자 새로고침 폴링으로 잡음

## Options Considered

| 옵션 | Pros | Cons | 비용 |
|---|---|---|---|
| A. 기존 ADR-001 유지 (큐 30석) | 결제 실패 시 자동 이전, 봇 30석 cap 1차 차단 | N=30 정당화 부담, 한국 실무 관행과 불일치, ADR-004 정합성 약함 | 구현 비용 중간, 정당화 비용 ↑ |
| B. **재고 카운터 + 새로고침 폴링** | 단순(N 정당화 불필요), 한국 실무 관행 일치, ADR-004 정합 | 봇 폴링 우위, 재고 풀림 ~ 다음 사용자 도착 사이 공백 시간 | 구현 비용 낮음, 봇 방어를 ADR-005에 의존 |
| C. 진입 큐(10) + 대기 큐(30) 분리 | FIFO 자동 이전 + 진입 명확 분리 | 사실상 ADR-001과 본질 같음, 복잡도만 ↑ | 구현 비용 높음 |

## Decision

**옵션 B 채택. 재고 카운터 기반 단일 진입 모델로 전환한다. 결제 실패/TTL 만료 시 재고가 INCR되며, *"재고 소진"* 응답을 받았던 사용자는 새로고침 폴링으로 재참여한다.**

### 데이터 모델

```
Redis Key: stock:accommodation:{id}
Type: Integer counter
Initial value: 10 (재고)
```

### 진입 로직 — Lua Script (ADR-002의 atomic 패턴 재사용)

```lua
-- KEYS[1]: stock key
-- ARGV[1]: hold key (e.g., "hold:user:{userId}:product:{productId}")
-- ARGV[2]: TTL seconds (300 = 5분)
-- Returns: 1 = entered, 0 = sold out

local stock = tonumber(redis.call('GET', KEYS[1]) or '0')
if stock <= 0 then
    return 0
end

-- 멱등성: 같은 사용자가 이미 hold 중이면 재진입 불가
if redis.call('EXISTS', ARGV[1]) == 1 then
    return 0
end

redis.call('DECR', KEYS[1])
redis.call('SET', ARGV[1], 'HOLD', 'EX', ARGV[2])
return 1
```

본 스크립트는 ADR-002에서 채택한 *"Lua Script atomic 처리"* 패턴을 재고 DECR에 적용한 것이다. 패턴 자체는 ADR-002의 결정에 그대로 의존한다.

### 결제 상태 전이

```
HOLD (재고 진입 직후, 결제 시작 전)
  ↓ POST /booking 호출
PG_PENDING (PG로 결제 요청 전송, 응답 대기 중)
  ↓ PG 응답 수신
COMPLETED 또는 FAILED
```

### 결제 TTL 동작 (5분 + PG 유예 60초)

```
00:00:00  진입 성공, 상태 HOLD, TTL 5분
00:04:30  사용자가 POST /booking 호출, 상태 PG_PENDING으로 전환
00:05:00  TTL 5분 도달
            상태 == HOLD: 즉시 회수 (재고 INCR)
            상태 == PG_PENDING: 60초 추가 유예
00:06:00  PG 유예 만료
            여전히 PG_PENDING: 강제 회수 + PG에 결제 취소 요청 (보상 트랜잭션)
            COMPLETED/FAILED: 정상 종료 흐름
```

PG 유예 60초의 근거: 한국 PG의 정상 응답 시간은 3~10초이며, 카드사 인증을 포함해도 30초 이내가 일반적이다. 60초는 충분한 마진이면서 무한 대기를 방지한다.

### 미결정 영역 — PG 응답 timeout 시나리오 (ADR-009 범위 밖)

본 ADR의 *"PG 유예 만료 시 강제 회수 + PG 취소 요청"*은 **PG가 응답을 보낸 상태**(우리가 PG_PENDING 상태로 인식하고 결과 대기 중)를 전제로 한다. 그러나 PG 호출 자체가 timeout으로 응답을 받지 못한 경우는 본 ADR의 결정으로 단순 처리되지 않는다.

미결정 시나리오 예시:
```
T+4:30  POST /booking 호출 → PG에 결제 요청 전송
T+4:35  PG 응답 timeout (네트워크 단절, PG 서버 무응답 등)
T+5:00  TTL 5분 도달, 상태 == PG_PENDING (응답 못 받음)
T+6:00  60초 유예 만료, 본 ADR에 따라 "강제 회수 + PG 취소 요청"

문제: 이 시점에 PG 측 실제 결제 상태를 모른다.
- PG가 결제 성공시킨 상태였다면 → 취소 요청은 정상 보상으로 작동
- PG가 결제 실패시킨 상태였다면 → 취소 요청은 의미 없음 (이미 실패)
- PG가 우리 호출 자체를 못 받았다면 → 취소할 결제가 없음
- PG 응답이 영원히 안 오면 → 무한 미결 상태
```

이 시나리오의 정확한 처리는 PG 결제 상태 조회 API를 통한 보상/확정 분기가 필요하며, **ADR-009에서 *"본 ADR의 범위 밖, 운영 절차"*로 명시적으로 분리**되어 있다. 본 ADR-008의 *"강제 회수 + PG 취소 요청"*을 PG timeout 시나리오에 그대로 적용하는 것은 부정확하다.

본 ADR이 결정한 것: PG가 응답을 보낸 후 60초 유예 동안 우리가 처리하지 못한 경우의 강제 회수 절차.
본 ADR이 결정하지 않은 것: PG 호출 자체가 timeout으로 응답을 받지 못한 경우의 reconciliation 절차.

### 재고 풀림 시 분배 정책

결제 실패 또는 TTL 만료로 재고가 INCR되면, 그 시점 이후 도착하는 요청 중 가장 먼저 Lua script를 통과한 사용자가 진입한다.

- *"재고 소진"* 응답을 받은 사용자는 클라이언트의 새로고침/재진입으로 다시 시도한다.
- 봇이 사람보다 빠르게 폴링할 수 있는 위협은 ADR-005(userId Token Bucket Rate Limit)와 본인인증(운영 환경 가정)으로 다룬다.
- 별도 대기열을 운영하지 않으므로 인터파크의 *"예매대기"* 같은 FIFO 자동 분배는 제공하지 않는다.

### 핵심 trade-off

- **Buy:** 모델 단순성 (재고 = 진입, N 정당화 불필요). 한국 실무 관행 정합 (인터파크 모델). ADR-004(약한 동등 시도 기회)의 5분 윈도우 내 지속적 보장. PG 응답 지연 시 자동 유예로 정상 사용자 보호.
- **Pay:** 결제 실패 직후 다음 사용자 폴링까지의 공백 시간 동안 재고가 비어있다. 봇이 사람보다 빠르게 폴링하면 재고를 선점할 가능성이 ADR-005만으로는 완전 차단되지 않는다. 자동 FIFO 이전이 없어 사용자가 능동적으로 새로고침해야 한다.

옵션 A(ADR-001 유지)를 선택하지 않은 이유: N=30의 정당화가 약하고, ADR-004와의 정합성이 떨어지며, 한국 실무 관행과 불일치한다.

옵션 C(진입 큐 + 대기 큐 분리)를 선택하지 않은 이유: 본질적으로 ADR-001과 같은 모델에 복잡도만 추가된다. ADR-001을 deprecated로 처리하기로 결정한 시점에 같은 흐름을 다른 형태로 살릴 이유가 없다.

## Consequences

**긍정 결과**
- 모델이 단일 카운터로 단순해진다. ZSET ticket 시스템 불필요.
- 5분 burst window 내내 모든 사용자가 새로고침으로 재참여 가능 — ADR-004 정합성 강화.
- 인터파크/콘서트 티켓 사용자에게 익숙한 흐름 — 별도 사용자 교육 불필요.
- ADR-002의 Lua atomic 패턴을 재사용해 구현 일관성 유지.
- PG 응답 지연 시 60초 유예로 카드 인증 진행 중 사용자 보호.

**부정 결과**
- 결제 실패 직후 ~ 다음 사용자 폴링 도착 사이의 공백 시간 동안 재고가 비어있다. 5초~수십 초의 비효율 가능.
- 봇이 다수 계정으로 분산 폴링 시 ADR-005만으로 완전 차단 어려움. 본인인증 가정에 의존.
- 새로고침 폴링 트래픽이 5분 윈도우 동안 평시보다 높게 유지된다. WebSocket 기반 실시간 알림이 근본 해결이지만 현 범위 over.
- PG 유예 60초 동안 재고가 묶여 있어, 다른 사용자에게 기회 이전이 늦어진다. PG 장애 시 영향 확대 가능.
- PG 유예 만료 후 강제 회수 시 PG에 보상 취소 요청을 보내야 한다. 보상 실패 시 PG 측 결제는 성공했는데 시스템에서는 재고 회수된 불일치 발생 가능 — ADR-010의 Outbox 패턴으로 대응.
- PG 호출 자체가 timeout으로 응답을 받지 못한 시나리오는 본 ADR의 단순 처리로 해결되지 않는다. PG 결제 상태 조회 API를 통한 보상/확정 분기가 필요하며, 이는 ADR-009에서 명시적으로 *"본 ADR의 범위 밖, 운영 절차"*로 분리되어 있다. 본 ADR의 *"강제 회수 + PG 취소 요청"*을 PG timeout 시나리오에 그대로 적용하면 PG 실제 상태와의 불일치가 발생할 수 있다.

**재검토 시점**
- 봇 폴링이 재고의 50% 이상 점유하는 패턴 관찰 시 (CAPTCHA, IP 기반 보조 Rate Limit 도입 검토)
- PG 유예 60초가 빈번히 trigger되는 경우 (PG 응답 시간 SLO 재평가 또는 유예 시간 조정)
- 5분 윈도우 동안 재고 회전 횟수가 예상보다 낮을 때 (자동 FIFO 이전 모델로 재전환 검토 — ADR-001 부활 고려)
- WebSocket 인프라 도입 가능 시 (폴링 → 푸시로 전환, 봇 우위 약화)
